# Publishing to Package Control

Sources: Package Control Official Docs (packagecontrol.io/docs), sublimehq/package_control_channel repository analysis, ST Community Docs (docs.sublimetext.io/guide/package-control/submitting.html)

Covers: Package Control channel submission process, JSON entry format, naming rules, versioning strategy, repository requirements, PR checklist, release messages, dependencies, testing with UnitTesting, GitHub Actions CI. Does NOT cover package file formats (see package-structure.md) or Python API (see plugin-api.md).

## Submission Process

### Step 1: Review Existing Packages

Search packagecontrol.io/search for similar packages. Improve existing packages
before creating a new one. Package Control rejects near-duplicates.

### Step 2: Name Your Package

| Rule | Rationale |
|------|-----------|
| No "Sublime" prefix | Redundant — all packages are for Sublime Text |
| CamelCase or spaces | `MyPackage` or `My Package`, not `my_package` |
| No dots | Breaks Python imports in ST3+ |
| No `/`, `<`, `>`, `:`, `"`, `\`, `\|`, `?`, `*` | Filesystem restrictions |
| ASCII only | Quick panel matching limitation |
| Language syntax: name after language | `Rust`, not `Rust Syntax Highlighting` |

### Step 3: Choose Hosting

GitHub or BitBucket (recommended). One package per repository.
Root of package must be root of repository.

Self-hosted `.sublime-package` files on HTTPS server is alternative
(rarely used).

### Step 4: Version with Semver Tags

Create semantic version git tags: `1.0.0`, `1.1.0`, `2.0.0`.
Branch-based releases are deprecated — no new packages accepted.

Tag prefix variants for multi-version support:
- `st3-1.0.0` → matches `"tags": "st3-"`
- `4107-1.0.0` → matches `"tags": "4107-"`

### Step 5: Prepare the Repository

- Remove all `.pyc` files
- Remove `package-metadata.json` (auto-generated by PC)
- Add `.no-sublime-package` if package includes executables
- Use `.gitattributes` with `export-ignore` for tests, docs, images
- Add `README.md` describing purpose and usage
- Add `LICENSE` file

### Step 6: Add to Channel

1. Fork `sublimehq/package_control_channel`
2. Add entry to alphabetical JSON in `repository/` (e.g., `repository/m.json`)
3. Keep entries alphabetically sorted within the file
4. Install ChannelRepositoryTools package
5. Run `ChannelRepositoryTools: Test Default Channel`
6. Submit PR — one package per PR

## Channel JSON Entry Format

### Minimal Entry (GitHub)

```json
{
    "name": "MyPackage",
    "details": "https://github.com/username/my-package",
    "releases": [
        {
            "sublime_text": "*",
            "tags": true
        }
    ]
}
```

### Full Entry with Metadata

```json
{
    "name": "MyPackage",
    "details": "https://github.com/username/my-package",
    "author": ["username"],
    "description": "A brief description of what the package does",
    "homepage": "https://example.com",
    "donate": "https://paypal.me/username",
    "labels": ["utilities", "productivity"],
    "releases": [
        {
            "sublime_text": ">=4000",
            "tags": true
        }
    ]
}
```

### Multi-Version Entry

```json
{
    "name": "MyPackage",
    "details": "https://github.com/username/my-package",
    "releases": [
        {"sublime_text": "3000 - 3999", "tags": "st3-"},
        {"sublime_text": ">=4000",      "tags": true}
    ]
}
```

### Platform-Restricted Entry

```json
{
    "releases": [
        {
            "sublime_text": "*",
            "platforms": ["osx", "linux"],
            "tags": true
        }
    ]
}
```

Platform values: `"windows"`, `"osx"`, `"linux"`.

### Field Reference

| Field | Required | Type | Notes |
|-------|----------|------|-------|
| `name` | Yes | string | Must follow naming rules above |
| `details` | Yes | string | GitHub/Bitbucket/GitLab URL |
| `releases` | Yes | array | At least one release object |
| `releases[].sublime_text` | Yes | string | `"*"`, `">=4000"`, `"3000 - 4000"` |
| `releases[].tags` | Yes | bool/string | `true` or prefix like `"st3-"` |
| `releases[].platforms` | No | array | Filter by OS |
| `labels` | No | array | Lowercase category strings |
| `author` | No | string/array | Defaults to repo owner |
| `description` | No | string | Short description |
| `readme` | No | string | URL to raw README |
| `donate` / `buy` | No | string | Donation/purchase URL |
| `previous_names` | No | array | For renamed packages |

`version`, `url`, and `date` sub-fields are NOT allowed in the
centralized channel. Only for self-hosted repositories.

## PR Checklist

The channel repository enforces this checklist:

- [ ] Author and/or maintainer of the package
- [ ] Read the submission docs
- [ ] Tagged a release with semver version number
- [ ] Repo has description and README
- [ ] Does not add context menu entries (unless contextually appropriate)
- [ ] Does not add default key bindings (provide Example.sublime-keymap instead)
- [ ] Commands available via command palette
- [ ] Settings open in split view (using `edit_settings` command)
- [ ] Syntax package does not bundle a color scheme
- [ ] Uses `.gitattributes` to exclude non-essential files

## Dependencies (dependencies.json)

Declare Package Control library dependencies:

```json
{
    "*": {
        ">=4000": ["typing_extensions"]
    }
}
```

Structure: `{ "<platform>": { "<st_version>": ["lib1", "lib2"] } }`

Platform: `"*"` (all), `"windows"`, `"osx"`, `"linux"`.

## Release Messages (messages.json)

Notify users on install and upgrade:

```json
{
    "install": "messages/install.txt",
    "1.0.0": "messages/1.0.0.txt",
    "2.0.0": "messages/2.0.0.txt"
}
```

Keep messages concise: what changed, migration steps, links.

## Testing with UnitTesting

The UnitTesting package (`SublimeText/UnitTesting`) runs Python unittest
inside Sublime Text.

### Setup

1. Create `tests/` directory with `test_*.py` files
2. Create `unittesting.json` at package root:

```json
{
    "tests_dir": "tests",
    "pattern": "test*.py",
    "deferred": true,
    "verbosity": 2,
    "reload_package_on_testing": true
}
```

### Running Locally

Command Palette > `UnitTesting: Test Current Package`.

### Writing Tests

```python
import sublime
from unittest import TestCase

class TestMyCommand(TestCase):
    def setUp(self):
        self.view = sublime.active_window().new_file()
        self.view.set_scratch(True)

    def tearDown(self):
        self.view.close()

    def test_insert_text(self):
        self.view.run_command("my_insert", {"text": "hello"})
        content = self.view.substr(sublime.Region(0, self.view.size()))
        self.assertEqual(content, "hello")
```

### GitHub Actions CI

```yaml
name: tests
on: [push, pull_request]

jobs:
  run-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: SublimeText/UnitTesting/actions/setup@v1
        with:
          sublime-text-version: 4
      - uses: SublimeText/UnitTesting/actions/run-tests@v1

  run-syntax-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: SublimeText/UnitTesting/actions/setup@v1
      - uses: SublimeText/UnitTesting/actions/run-syntax-tests@v1
```

### Available CI Actions

| Action | Purpose |
|--------|---------|
| `SublimeText/UnitTesting/actions/setup@v1` | Install ST + UnitTesting + PC |
| `SublimeText/UnitTesting/actions/run-tests@v1` | Run Python unit tests |
| `SublimeText/UnitTesting/actions/run-syntax-tests@v1` | Run syntax test files |
| `SublimeText/UnitTesting/actions/run-color-scheme-tests@v1` | Run color scheme tests |

### Setup Action Inputs

| Input | Default | Purpose |
|-------|---------|---------|
| `sublime-text-version` | 4 | ST version (3 or 4) |
| `sublime-text-arch` | x64 | Architecture |
| `install-package-control` | true | Install PC |
| `extra-packages` | | Additional packages to install |
